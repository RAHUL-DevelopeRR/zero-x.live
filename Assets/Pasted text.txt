Short answer
VLC Media Player cannot generate offline AI subtitles by itself.
VLC/libVLC is excellent for playback, codec support, timing, subtitle rendering, track handling, etc. But VLC does not include a built-in offline speech-to-text engine that can transcribe arbitrary video/audio into subtitles.

You can use VLC/libVLC as the media engine, but subtitle generation must come from something like:

whisper.cpp
faster-whisper / CTranslate2
Vosk/Kaldi
NVIDIA/Riva-style local ASR
other local ASR engines
So your Lumina approach — VLC for playback + Whisper/FFmpeg for offline subtitle generation — is the correct direction.

But the claim:

“Any language, any slang, >95% accurate”

is not realistic.

Even Whisper large models are not consistently >95% accurate across all languages, accents, dialects, slang, noisy audio, overlapping speech, music, bad microphones, movies with effects, low-resource languages, etc. For clean English or high-resource languages, you may get excellent results. For “any language and slang”, no.

Also: I cannot live-browse the web from here. This analysis is based on known public technical behavior of VLC/libVLC, FFmpeg, Whisper/whisper.cpp, Argos, LibreTranslate, Ollama, JavaFX/vlcj, and common production architecture constraints.

1. Can VLC be used for offline subtitle generation?
Regular VLC desktop app
No, not by itself.

VLC can:

play almost any media format
load .srt, .ass, .vtt, etc.
render subtitles
shift subtitle delay
select subtitle tracks
extract/stream/transcode media in some workflows
use extensions/scripts for things like subtitle search/download
But VLC does not ship with a full offline ASR pipeline.

There have been plugins/extensions/projects that integrate Whisper-like functionality with VLC, but those are external integrations, not core VLC behavior.

libVLC inside your own app
Yes, but only as the media/playback layer.

This is the right usage:


Video playback/timing/rendering  -> VLC/libVLC  
Audio extraction/decoding        -> FFmpeg or libav  
Speech-to-text                   -> whisper.cpp / faster-whisper  
Subtitle storage                 -> SRT/VTT/DSRT/SQLite  
Subtitle display                 -> JavaFX overlay or VLC subtitle track  
Your architecture is therefore conceptually valid.

VLC plugin approach
Technically possible, but I would not recommend it for your product.

A VLC plugin would likely mean:

native C/C++ work
VLC plugin ABI constraints
harder UI
harder packaging
harder cross-platform support
harder model management
harder async job management
ugly lifecycle/threading problems
For a shippable product, your current external app using libVLC is much better.

2. The >95% accurate in any language/slang claim is the biggest problem
This is product/marketing suicide if you ship it that way.

A more honest claim would be:

“Generates offline AI subtitles with high accuracy on clear speech, with quality depending on model size, language, audio quality, accent, and hardware.”

Whisper is very good, but not magic.

What affects accuracy?
Factor	Impact
Model size	tiny/base are fast but weaker. medium/large are better but slower/heavier.
Language	English and high-resource languages are much better than low-resource languages.
Audio quality	Noise, music, compression, echo, bad microphones reduce accuracy.
Speech style	Slang, dialects, code-switching, mumbling, fast speech reduce accuracy.
Multiple speakers	Overlap causes errors. Whisper does not solve diarization by itself.
Domain vocabulary	Names, technical terms, anime/game terms, medical/legal terms can fail.
Chunking	Bad chunk boundaries cause missing/duplicated words.
Translation	Translation quality is a second separate error source.
Realistic expectations
Approximate practical expectations:

Scenario	Expected result
Clean English podcast/interview, medium/large	Very good, sometimes near professional
Clean high-resource language	Good to very good
Movie with background music/effects	Mixed
Heavy slang/dialect	Variable
Low-resource language	Can be poor
Noisy street/interview audio	Often poor
Overlapping speakers	Bad without diarization
Translation to another language	Depends heavily on translation engine
So: offline subtitles are feasible. Universal >95% accuracy is not.

What is good
The core architecture is sane:


JavaFX UI  
  -> PlayerController  
    -> VlcEngine/libVLC  
  
AI Subtitle Pipeline  
  -> FFmpeg audio extraction  
  -> whisper.cpp transcription  
  -> progressive subtitle storage

That is a reasonable split.

The strongest parts are:

libVLC only handles media playback
FFmpeg handles audio extraction
Whisper handles ASR
Subtitle state is separated from player state
Progressive subtitle generation is supported
UI is decoupled using JavaFX properties
Background processing avoids blocking the UI
This is not a toy design. It is a credible engineering direction.

Biggest problems
1. Fixed 30-second chunks are crude
Fixed chunking is simple, but it creates problems:

sentences split across chunks
missing words at boundaries
duplicated words at boundaries
context loss
worse punctuation
worse translation
timing drift
Better approach:


Decode audio -> VAD segmentation -> overlap windows -> Whisper -> stitch/dedupe -> subtitle cues  
Use voice activity detection instead of blindly slicing every 30 seconds.

2. Running FFmpeg and Whisper as CLI processes per chunk is fragile
It works for a prototype.

For production, it creates problems:

process startup overhead
temp file explosion
CPU oversubscription
orphaned processes on cancel
stdout/stderr deadlocks if streams are not drained
hard timeout handling
inconsistent environment/PATH issues
difficult packaging
For v1 it is acceptable. For v2, use a worker service or embedded library.

3. ConcurrentSkipListMap<Long, DsrtCue> has edge cases
Keying only by startTimeMs can break if:

two cues start at same millisecond
overlapping cues exist
reprocessing a chunk creates replacement cues
translated and original cues coexist
provisional/final cues need versioning
Better key:


(startTimeMs, cueId)  
or:


NavigableMap<Long, List<DsrtCue>>  
or use SQLite:


cues(job_id, cue_id, start_ms, end_ms, text, status, version, chunk_id)  
4. .dsrt is useful, but do not call it “a new standard”
Custom formats rarely become standards.

.dsrt is good as an internal progressive subtitle/job format. But externally you should export:

.srt
.vtt
.ass
For production I would probably use:


SQLite WAL database for live/progressive state  
Export SRT/VTT/ASS when needed  
JSON is fine for a small prototype, but repeated atomic rewrites of a growing JSON file become inefficient and can cause concurrency pain.

5. Translation pipeline is overambitious
The orchestration system is impressive, but translation is a second product.

ASR alone is hard. ASR + translation + verification + multiple providers + offline model management is much harder.

For shipping:

ship transcription first
add export
add offline translation for selected language pairs
then add advanced orchestration
Do not make translation block your MVP.

What is better in the second version

But it also has more red flags
1. “First 30 seconds in 3 seconds” is not guaranteed
This depends heavily on:

CPU/GPU
model size
quantization
audio length
language
beam size
thermal throttling
other running processes
With tiny, yes, maybe on a decent machine.

But tiny is not where your best accuracy comes from.

So this creates a bad contradiction:


Fast first subtitles -> tiny/fast -> lower accuracy  
High accuracy         -> medium/large/best -> slower  
You cannot promise both universally.

2. Priority chunk with tiny then reprocessing with better model creates UX problems
If the user first sees:

“I went to the market.”

Then later the better model changes it to:

“I wouldn’t go to the market.”

That is not just a minor correction. It can change meaning.

You need to mark cues as:


PROVISIONAL  
FINAL  
CORRECTED  
FAILED  
And the UI should indicate draft subtitles.

3. Daemon threads do not solve process cleanup
This line is dangerous:

“Daemon threads ensure the JVM can exit cleanly.”

Daemon Java threads may stop, but external child processes may survive unless explicitly killed.

You need:

process registry
cancellation tokens
ProcessHandle.descendants()
timeout handling
kill gracefully, then forcibly
temp file cleanup
shutdown hook
4. CopyOnWriteArrayList is not ideal for frequently updated chunks
CopyOnWriteArrayList is good when reads dominate and writes are rare.

Chunk status updates are writes. If you update progress frequently, use:

ConcurrentHashMap<Integer, ChunkState>
AtomicReferenceArray<ChunkState>
immutable events through an event bus
JavaFX observable model updated only on FX thread
5. Platform.runLater() can flood the UI queue
You mention throttling to 50ms, which is good.

But for production, I would coalesce updates:


Native VLC event thread updates atomic state  
JavaFX AnimationTimer reads latest state at 30/60 FPS  
This prevents UI event queue buildup.

6. Argos/Libre/Ollama translation is not universally offline or universally good
Argos is offline after language packs are installed, but language coverage varies.
LibreTranslate can be local, but often people use a server/API.
Ollama is local, but LLM translation quality and speed vary wildly.
Verification by another model sounds nice but can introduce new errors.
Translation should be positioned as “best effort”.

5. Major architectural fixes I would make
Fix 1: Replace fixed 30-second chunking with VAD-based segmentation
Current:


0-30s  
30-60s  
60-90s  
...  
Better:


audio -> VAD -> speech segments -> merge into 15-45s windows -> overlap -> transcribe -> stitch  
Use:

WebRTC VAD
Silero VAD
whisper.cpp VAD support if suitable
FFmpeg silence detection as a fallback
Benefits:

fewer cut sentences
better timestamps
better context
better translation
less duplicated/missing text
Fix 2: Add overlap and deduplication
Every segment should have overlap:


Chunk 1: 00:00 - 00:32  
Chunk 2: 00:28 - 01:02  
Chunk 3: 00:58 - 01:32  
Then deduplicate repeated words/cues in the overlap region.

Without overlap, chunk boundaries will hurt you.

Fix 3: Use a job database instead of only JSON
For production, I would use SQLite:


jobs  
  id  
  media_hash  
  media_path  
  duration_ms  
  source_language  
  target_language  
  model  
  status  
  created_at  
  updated_at  
  
segments  
  id  
  job_id  
  start_ms  
  end_ms  
  status  
  attempt_count  
  error  
  
cues  
  id  
  job_id  
  segment_id  
  start_ms  
  end_ms  
  text  
  translated_text  
  confidence  
  status  
  version  
Use WAL mode.

Benefits:

resumable
crash-safe
queryable
no giant JSON rewrite
easier migration
easier cache by media hash
easier export
You can still export .dsrt if you want.

Fix 4: Make .dsrt an internal/cache format, not the primary product promise
Good external formats:

.srt
.vtt
.ass
Internal progressive format:

SQLite
JSONL event log
.dsrt
If you keep .dsrt, define it as:

“Lumina’s internal dynamic subtitle project format.”

Not:

“A new standard.”

Fix 5: Separate draft and final subtitle passes
Use two modes:

Draft mode
Fast, immediate:


small model / quantized model / low beam / current segment first  
UI label:


AI subtitles: Draft  
Final mode
Slower, accurate:


medium/large model / better beam / VAD segments / context stitching  
UI label:


AI subtitles: Final  
This solves the speed-vs-quality contradiction.

Fix 6: Do not run unlimited Whisper workers
Whisper is CPU/GPU heavy. Parallelizing many chunks can make everything slower.

Bad:


4 chunks -> 4 whisper processes -> 4 processes fighting for CPU/GPU/RAM  
Better:


Audio extraction pool: 2 workers  
ASR worker: 1 worker per GPU/CPU profile  
Translation pool: 1-3 workers  
For CPU inference, often one Whisper process using many threads is better than many Whisper processes fighting.

Fix 7: Better cancellation
Every job needs:


cancel()  
pause()  
resume()  
killProcessTree()  
cleanupTempFiles()  
markIncompleteSegments()  
External process cleanup is mandatory.

Fix 8: Add content hashing and caching
If a user opens the same media again, subtitles should load instantly.

Use:


hash = size + modified time + partial content hash  
or stronger:


SHA-256 of media file  
For large files, use a hybrid hash.

Cache location:


~/.lumina/cache/subtitles/{mediaHash}/job.sqlite  
~/.lumina/cache/subtitles/{mediaHash}/final.srt

8. Recommended ASR pipeline
Current pipeline

Split into 30s chunks  
  -> FFmpeg extract WAV  
  -> whisper.cpp  
  -> insert cues  
Better pipeline

Media file  
  -> Decode audio to 16k mono PCM  
  -> Voice activity detection  
  -> Segment speech intelligently  
  -> Add 1-2s overlap  
  -> Transcribe segment  
  -> Stitch/deduplicate  
  -> Normalize timestamps  
  -> Store draft cues  
  -> Optional final correction pass  
  -> Export SRT/VTT  
Why this is better
Current fixed chunks	VAD-based segmentation
cuts sentences	preserves speech units
boundary errors	fewer boundary errors
worse translation	better translation context
duplicate/missing words	easier dedupe
arbitrary 30s windows	adaptive to actual speech

11. VLC vs FFmpeg for audio extraction
Use VLC for playback.

Use FFmpeg or libav for audio extraction.

Do not try to make VLC do the ASR audio pipeline unless you have a very good reason.

Why FFmpeg is better here
predictable CLI behavior
great seeking/extraction
sample rate conversion
channel conversion
mature
easy to test independently
already standard in media pipelines
Your FFmpeg command is mostly right:


ffmpeg -ss <start> -i <input> -t <duration> \  
  -vn -acodec pcm_s16le -ar 16000 -ac 1 -y output.wav  
But be careful: -ss before -i is faster but can be less sample-accurate. For subtitles, this may matter.

For final pass, consider accuracy over speed.

Final verdict
Your idea is good. The architecture direction is mostly correct.

But:

VLC does not generate subtitles.
VLC should remain the playback engine, not the ASR engine.
Whisper/FFmpeg/ASR pipeline should be a separate subtitle engine.
The >95% any language/slang claim is false and should be removed.
The current design is strong as a prototype but too overclaimed for production.
Fixed 30-second chunking should be replaced with VAD + overlap + stitching.
Use SQLite/job cache instead of relying only on custom JSON .dsrt.
Ship transcription first. Translation later. Verification much later.
Best shippable product positioning:

Lumina Player: a privacy-first desktop media player that generates offline AI subtitles locally using Whisper-based models, with progressive subtitle display and export to standard subtitle formats.

That is feasible. That is shippable. That is valuable.
